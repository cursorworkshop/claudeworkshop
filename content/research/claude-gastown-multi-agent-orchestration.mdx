# Claude Gas Town: Multi-Agent Orchestration for Claude Code

We've been working on something we're excited to share: **Claude Gas Town**â€”a Claude Code implementation of Steve Yegge's Gas Town multi-agent orchestrator.

Built in collaboration with the Claude engineering team, Claude Gas Town brings persistent work tracking and multi-agent coordination to Claude Code workflows.

## What Is Gas Town?

Gas Town is a workspace manager that coordinates multiple AI coding agents working on different tasks. Instead of losing context when agents restart, Gas Town persists work state in git-backed hooks, enabling reliable multi-agent workflows.

The original Gas Town was built for Claude Code. We've adapted it for Claude Code, maintaining the same architecture while targeting the Claude Code ecosystem.

## The Core Problem It Solves

| Challenge                       | Gas Town Solution                            |
| ------------------------------- | -------------------------------------------- |
| Agents lose context on restart  | Work persists in git-backed hooks            |
| Manual agent coordination       | Built-in mailboxes, identities, and handoffs |
| 4-10 agents become chaotic      | Scale comfortably to 20-30 agents            |
| Work state lost in agent memory | Work state stored in Beads ledger            |

## How It Works

### Core Concepts

**The Mayor** â€” Your primary AI coordinator. The Mayor is a Claude Code agent with full context about your workspace, projects, and agents. Start hereâ€”just tell the Mayor what you want to accomplish.

**Town** â€” Your workspace directory (e.g., `~/gt/`). Contains all projects, agents, and configuration.

**Rigs** â€” Project containers. Each rig wraps a git repository and manages its associated agents.

**Crew Members** â€” Your personal workspace within a rig. Where you do hands-on work.

**Polecats** â€” Ephemeral worker agents that spawn, complete a task, and disappear.

**Hooks** â€” Git worktree-based persistent storage for agent work. Survives crashes and restarts.

**Convoys** â€” Work tracking units. Bundle multiple beads (tasks) that get assigned to agents.

**Beads** â€” Git-backed issue tracking system that stores work state as structured data.

### The Workflow

1. **Create work as beads** â€” Define tasks using the Beads system
2. **Group into convoys** â€” Track related work together
3. **Assign to agents** â€” Use `gt sling` to dispatch work
4. **Monitor progress** â€” Track through convoy status
5. **Work persists** â€” State survives agent restarts via git hooks

## Installation

### Prerequisites

- Go 1.23+
- Git 2.25+ (for worktree support)
- beads (bd) 0.47.0+
- tmux 3.0+ (recommended)
- Claude Code

### Setup

```bash
# Install Gas Town and Beads
go install github.com/claudeworkshop/claude-gastown/cmd/gt@latest
go install github.com/steveyegge/beads/cmd/bd@latest
export PATH="$PATH:$HOME/go/bin"

# Create workspace
gt install ~/gt --git
cd ~/gt

# Add a project
gt rig add myproject https://github.com/you/repo.git

# Create crew workspace
gt crew add yourname --rig myproject

# Verify setup
gt doctor
```

## Quick Start

```bash
# Navigate to your rig
cd ~/gt/myproject

# Create beads for your tasks
bd create "Add user authentication"
bd create "Fix login page styling"
bd create "Write integration tests"

# Create a convoy tracking your beads
gt convoy create "Auth Feature" gt-abc gt-def gt-ghi --notify

# Assign work to agents
gt sling gt-abc myproject

# Monitor progress
gt convoy list
```

## Key Features

### Workflow Durability

The most interesting aspect isn't the parallelismâ€”it's how Gas Town handles durability.

Each agent has a "hook" where you assign work. If Claude Code crashes or runs out of context, the workflow state lives in Git through the Beads system. A new agent picks up exactly where the last one stopped. No lost progress, no re-explaining context.

### Multi-Agent Coordination

Gas Town coordinates multiple agents working on different tasks simultaneously. The Mayor agent dispatches work, and agents coordinate through git-backed hooks.

### Persistent State

All work state is stored in git-backed hooks. This means:

- Work survives agent crashes
- You can resume from any point
- Full history of changes
- Easy rollback capability

### Formulas

- `shiny` â€” Full feature workflow (design â†’ implement â†’ review â†’ test â†’ submit)

- `design` â€” Design-only phase

## How This Relates to Our Methodology

At Claude Workshop, we teach the **Delegate, Review, Own** framework. Claude Gas Town fits this model perfectly:

**Delegate**: Gas Town handles the orchestration of boilerplate, standard implementations, and repetitive tasks across multiple agents. You describe what needs to happen; the agents figure out how.

**Review**: The Refinery role provides automated review, but humans still own the final verification. Gas Town produces work faster, which means more time for meaningful review.

**Own**: Strategic decisionsâ€”what to build, architectural choices, technology selectionâ€”remain human responsibilities. Gas Town amplifies execution but doesn't replace judgment.
